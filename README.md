# Проект "Параметры"

Проект представляет собой веб-приложение для управления системными и пользовательскими параметрами. Цель — создать интуитивно понятный интерфейс для просмотра, фильтрации и редактирования различных конфигурационных настроек.

Приложение разрабатывается с использованием HTML, CSS и JavaScript. Данный репозиторий используется в учебных целях для моделирования процесса разработки по методологии Scrum.

В рамках лабораторной работы мы имитируем один спринт, используя GitHub Issues для управления бэклогом продукта и отслеживания прогресса.

## Sprint 1 Review

**Завершённые задачи (Closed Issues):**
*   **#1: Создать базовую HTML-структуру главной страницы.** Доставлен файл `index.html` с семантической разметкой.
*   **#2: Добавить таблицу для отображения списка параметров.** Таблица с тестовыми данными добавлена на страницу.
*   **#3: Реализовать базовые CSS-стили.** Подключён файл `styles.css`, задано оформление для страницы и таблицы.
*   **#6: Написать документацию по установке и запуску.** Создан файл `INSTALL.md` с инструкцией для новых разработчиков.

**Доставленный результат по итогам спринта:**
В репозитории находится работающее веб-приложение с главной страницей, таблицей параметров и базовым дизайном. Любой разработчик может склонировать проект и сразу открыть его в браузере, о чём написана документация. Реализован фундамент для добавления будущей функциональности.

**Не завершена в спринте:**
*   **#5: Реализовать фильтрацию параметров по названию.** Задача частично выполнена (добавлено поле поиска), но JavaScript-код для динамической фильтрации не был полностью реализован и протестирован. Задача остаётся открытой и будет перенесена в следующий спринт с высоким приоритетом.

## Sprint 1 Retrospective

**Что прошло хорошо (What went well):**
1.  **Чёткий бэклог:** Заранее подготовленные Issues с понятным описанием и критериями готовности (DoD) сильно ускорили планирование и выполнение задач.
2.  **Визуализация прогресса:** Работа с метками (`todo`, `in-progress`, `closed`) в GitHub Issues позволила всем "членам команды" (в рамках симуляции) наглядно видеть, что делается и что уже готово.
3.  **Достигнута основная цель спринта:** Несмотря на незавершённую задачу, мы успешно создали рабочее ядро приложения (верстка, данные, стили).

**Что можно улучшить (What could be improved):**
1.  **Ошибки в оценке:** Задача по фильтрации (#5) была оценена в 5 story points, но на практике оказалась сложнее из-за необходимости детальной работы с DOM на JavaScript. Её следовало разбить на подзадачи или оценить выше.
2.  **Нехватка времени на тестирование:** Из-за того, что мы взяли в спринт слишком много "средних" по сложности задач (верстка таблицы, стили), на отладку JavaScript почти не осталось времени.
3.  **Отсутствие ежедневных стендапов (в симуляции):** В реальной команде ежедневное обсуждение прогресса помогло бы раньше выявить сложности с задачей #5.

**План улучшений на следующий спринт:**
1.  **Более детальное планирование:** На следующем планировании спринта проводить короткое техническое обсуждение каждой сложной задачи перед её включением.
2.  **Буфер на непредвиденное:** Закладывать в спринт на 1-2 задачи меньше от максимального расчётного объёма, чтобы оставалось время на доработки и исправление багов.
3.  **Разбивка больших задач:** Задачи с оценкой в 5+ story points обязательно разбивать на более мелкие подзадачи (sub-issues) перед началом работы.
